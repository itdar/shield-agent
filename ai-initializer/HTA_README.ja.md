# AIエージェント管理システム — 人間のためのガイド

## 始めるには？

> **トークン使用量の注意** — 初期セットアップでは上位モデルを使用してプロジェクト全体を分析し、複数のファイル（AGENTS.md、.ai-agents/context/、.ai-agents/skills/、.ai-agents/roles/）を生成します。プロジェクト規模により数万トークン以上消費する場合があります。これは1回限りのコストであり、以降のセッションは構築済みコンテキストを読み込み即座に開始できます。

```bash
# 1. AIにHOW_TO_AGENTS.mdを読ませれば自動的に設定される

# オプション A：英語（推奨 — トークンコスト削減、AI性能最適）
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "Read HOW_TO_AGENTS.md and generate AGENTS.md tailored to this project"

# オプション B：ユーザーの言語（AGENTS.mdを人が直接編集する場合推奨）
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "HOW_TO_AGENTS.mdを読んで、このプロジェクトに合わせてAGENTS.mdを生成せよ（日本語で）"

# 推奨：--dangerously-skip-permissions で中断なしの自律セットアップ
# 推奨：--model claude-opus-4-6（またはそれ以降）で最良の結果

# 2. 生成されたエージェントで作業開始
./ai-agency.sh

# 3. 以降のセッションからは自動的にすぐ作業可能！
```

> この文書は**人間が読んで理解するための**文書です。
> AIが実行する指示書（HOW_TO_AGENTS.md）がなぜ存在し、どのような原理で動作し、
> あなたの開発ワークフローでどのような役割を果たすのかを説明します。

## なぜこのようなシステムが必要なのか？

### 問題：AIは毎回記憶を失う

```
 セッション1                セッション2                セッション3
┌──────────┐             ┌──────────┐             ┌──────────┐
│ AIがコード │             │ AIが再び  │             │ また最初から│
│ 全体を読む │  セッション終了│ 全体を読む │  セッション終了│ 全体を読む │
│ (30分)    │ ──────→    │ (30分)    │ ──────→    │ (30分)    │
│ 作業開始   │ 記憶消滅！   │ 作業開始   │ 記憶消滅！   │ 作業開始   │
└──────────┘             └──────────┘             └──────────┘
```

AIエージェントはセッションが終わるとすべてを忘れます。
毎回プロジェクト構造を把握し、APIを分析し、コンベンションを理解するのに時間を費やします。

### 解決策：AIに「脳」を事前に作っておく

```
 セッション開始
┌──────────────────────────────────────────────────┐
│                                                  │
│  AGENTS.mdを読み込む（自動）                        │
│       │                                          │
│       ▼                                          │
│  "私はdoppel-apiのバックエンド専門家"                 │
│  "コンベンション: Conventional Commits, TypeScript strict" │
│  "禁止: 他のサービスの修正、シークレットのハードコーディング" │
│       │                                          │
│       ▼                                          │
│  .ai-agents/context/ ファイルをロード（5秒）                │
│  "API 20個、エンティティ 15個、イベント 8個 把握完了"   │
│       │                                          │
│       ▼                                          │
│  即座に作業開始！                                   │
│                                                  │
└──────────────────────────────────────────────────┘
```

## 核心原理：3層構造

```
                    あなたのプロジェクト
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼

     ┌──────────┐  ┌──────────┐  ┌──────────┐
     │ AGENTS.md│  │.ai-agents/context│  │.ai-agents/skills│
     │          │  │          │  │          │
     │ 人格     │  │ 知識     │  │ 行動     │
     │ "私は    │  │ "この    │  │ "開発する│
     │  誰なのか"│  │  サービス │  │  ときは  │
     │          │  │  のAPIは │  │  こうする"│
     │ + ルール │  │  こうだ"  │  │          │
     │ + 権限   │  │          │  │ + デプロイ│
     │ + パス   │  │ + ドメイン│  │ + レビュー│
     └──────────┘  └──────────┘  └──────────┘
         エントリ      記憶ストア     ワークフロー標準
         ポイント
```

### 1. AGENTS.md — "私は誰なのか"

各ディレクトリに配置されるエージェントの**アイデンティティファイル**です。

```
プロジェクト/
├── AGENTS.md                  ← PM: 全体を調整するリーダー
├── apps/
│   └── doppel-api/
│       └── AGENTS.md          ← API専門家: このサービスのみ担当
├── infra/
│   ├── AGENTS.md              ← SRE: インフラ全体を管理
│   └── monitoring/
│       └── AGENTS.md          ← モニタリング専門家
└── configs/
    └── AGENTS.md              ← 設定管理者
```

まるで**チーム組織図**のようなものです：
- PMが全体を見て作業を分配
- 各メンバーは自分の領域だけを深く理解
- 他のチームの仕事は直接行わず依頼する

### 2. `.ai-agents/context/` — "何を知っているか"

AIが毎回コードを読まなくても済むように**コア知識を事前に整理**したフォルダです。

```
.ai-agents/context/
├── domain-overview.md     ← "このサービスは注文管理を担当し..."
├── data-model.md          ← "Order, Payment, Deliveryエンティティがあり..."
├── api-spec.json          ← "POST /orders, GET /orders/{id}, ..."
└── event-spec.json        ← "order-createdイベントを発行し..."
```

**例え：** 新入社員に渡すオンボーディング文書と同じです。
「私たちのチームが何をしているか、DB構造はどうなっているか、APIには何があるか」を一度整理しておけば、
毎回説明する必要がありません。

### 3. `.ai-agents/skills/` — "どのように仕事をするか"

繰り返される作業を標準化した**ワークフローマニュアル**です。

```
.ai-agents/skills/
├── develop/SKILL.md       ← "機能開発: 分析 → 設計 → 実装 → テスト → PR"
├── deploy/SKILL.md        ← "デプロイ: タグ → 申請書 → 検証"
└── review/SKILL.md        ← "レビュー: セキュリティ・パフォーマンス・テストチェックリスト"
```

**例え：** チームの作業マニュアルです。
「PRを出すときはこのチェックリストを確認して」のようなことをAIにも従わせます。

## グローバルルールはどのように管理するか？

**継承パターン**を使用します。一箇所に書けば、下位に自動適用されます。

```
ルート AGENTS.md ──────────────────────────────────────────
│ Global Conventions:
│  - コミット: Conventional Commits (feat:, fix:, chore:)
│  - PR: テンプレート必須、1名以上のレビュー
│  - ブランチ: feature/{ticket}-{desc}
│  - コード: TypeScript strict, single quotes
│
│     自動継承                      自動継承
│     ┌──────────────────┐       ┌──────────────────┐
│     ▼                  │       ▼                  │
│  apps/api/AGENTS.md    │    infra/AGENTS.md       │
│  (追加ルールのみ記載)     │    (追加ルールのみ記載)     │
│  "このサービスは          │    "Helm values 変更時     │
│   Python 使用"          │     Ask First"           │
│     (TypeScript の代わり)│                          │
└─────────────────────────┴──────────────────────────
```

**利点：**
- コミットルールを変えたい場合？ → ルートの一箇所だけ修正
- 新しいサービスを追加した場合？ → 自動的にグローバルルールが適用
- 特定のサービスだけ異なる設定にしたい場合？ → そのサービスのAGENTS.mdでオーバーライド

## 何を書き、何を書かないべきか？

ETH Zurich の研究（2026年）によると、AIがすでに推論できる内容を文書に書くと、
むしろ**成功率が低下しコストが20%増加**します。

```
                書くもの                          書かないもの
     ┌─────────────────────────┐     ┌─────────────────────────┐
     │                         │     │                         │
     │  "コミットはfeat:形式で"    │     │  "src/フォルダにソースがある" │
     │  AIが推論できないもの       │     │  AIがlsですぐ確認できる     │
     │                         │     │                         │
     │  "mainへの直接push禁止"   │     │  "Reactはコンポーネントベース"│
     │  チームルールでコードにない   │     │  公式ドキュメントに既にある   │
     │                         │     │                         │
     │  "デプロイ前にQAチーム承認必須"│     │  "このファイルは100行だ"    │
     │  プロセスなので推論不可     │     │  AIが直接読めばよい         │
     │                         │     │                         │
     └─────────────────────────┘     └─────────────────────────┘
             AGENTS.mdに記載                    記載禁止！
```

**ただし例外があります：**「推論は可能だが、毎回行うとコストが高すぎるもの」

```
  例: 全API一覧（20個のファイルをすべて読む必要がある）
  例: データモデル関係図（10個のファイルに散在している）
  例: サービス間呼び出し関係（コード + インフラ両方の確認が必要）

  → これらは.ai-agents/context/に事前に整理しておく！
  → AGENTS.mdには「ここに行けばある」というパスだけ書く
```

## セッションランチャースクリプト

すべてのエージェントが設定されたら、希望のエージェントを選んですぐにセッションを開始できます。

```bash
$ ./ai-agency.sh

=== AI Agent Sessions ===
Project: /Users/nhn/src/doppel-deploy
Found: 8 agent(s)

  1) [PM] doppel-deploy
     Path: ./AGENTS.md
     このプロジェクトのPMエージェント。全体構造を把握し作業を委任する。

  2) doppel-api
     Path: apps/dev/doppel-api/AGENTS.md
     doppel-apiサービスのK8sマニフェスト管理専門家。

  3) monitoring
     Path: infra/monitoring/AGENTS.md
     Prometheus + Grafanaモニタリングスタックの SRE専門家。

  ...

Select agent (number): 2

=== AI Tool ===
  1) claude
  2) codex
  3) print

Select tool: 1

→ claudeセッションがdoppel-apiディレクトリで開始
→ エージェントが自動的にAGENTS.mdと.ai-agents/context/をロード
→ 即座に作業可能！
```

**並列実行（tmux）：**

```bash
$ ./ai-agency.sh --multi

Select agents: 1,2,3   # PM + API + モニタリング同時実行

→ tmuxセッション3つが開く
→ 各ウィンドウで異なるエージェントが独立して作業
→ Ctrl+B Nでウィンドウ切り替え
```

## 成果物一覧

このシステムで作成されるファイルとそれぞれの用途：

| ファイル | 対象 | 用途 |
|---|---|---|
| `HOW_TO_AGENTS.md` | AI | エージェントが読んで実行するメタ指示書 |
| `HOW_TO_AGENTS_PLAN.md` | 人間/AI | 設計計画書（なぜこの構造なのかの背景） |
| `README.md` | 人間 | この文書。人間が理解するためのガイド |
| `ai-agency.sh` | 人間 | エージェント選択 → AIセッション開始ランチャー |
| `AGENTS.md`（各ディレクトリ） | AI | ディレクトリごとのエージェントアイデンティティ + ルール |
| `.ai-agents/context/*.md/json` | AI | 事前に整理されたドメイン知識 |
| `.ai-agents/skills/*/SKILL.md` | AI | 標準化された作業ワークフロー |
| `.ai-agents/roles/*.md` | AI/人間 | 役割別コンテキストローディング戦略 |

## 核心的な例え

```
              従来の開発チーム            AIエージェントチーム
              ────────────             ────────────────
 リーダー     PM（人間）                ルート AGENTS.md（PMエージェント）
 メンバー     開発者N名                 各ディレクトリのAGENTS.md
 オンボーディング文書  Confluence/Notion  .ai-agents/context/
 作業マニュアル 　チームWiki              .ai-agents/skills/
 役割定義     職級/R&R文書              .ai-agents/roles/
 チームルール   チームコンベンション文書    Global Conventions（継承）
 出社        オフィス到着               セッション開始 → AGENTS.mdロード
 退社        退社（記憶保持）            セッション終了（記憶消滅！）
 翌日の出社   記憶あり                  .ai-agents/context/ロード（記憶復元）
```

**核心的な違い：** 人間は退社しても記憶を持っていますが、AIは毎回忘れます。
だから`.ai-agents/context/`が存在するのです ── AIの**長期記憶**の役割を果たします。
