# Система управления AI-агентами — Руководство для людей

## С чего начать?

> **Предупреждение: Высокий расход токенов** — Первоначальная настройка использует модель высшего уровня для анализа всего проекта и создания нескольких файлов (AGENTS.md, .ai-agents/context/, .ai-agents/skills/, .ai-agents/roles/). Ожидайте значительный расход токенов (десятки тысяч и более в зависимости от размера проекта). Это разовые затраты — последующие сессии загружают готовый контекст и стартуют мгновенно.

```bash
# 1. Дайте AI прочитать HOW_TO_AGENTS.md — он настроит всё сам

# Вариант A: Английский (рекомендуется — меньше затрат токенов, лучшая производительность ИИ)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "Read HOW_TO_AGENTS.md and generate AGENTS.md tailored to this project"

# Вариант B: Ваш язык (рекомендуется если люди редактируют AGENTS.md напрямую)
claude --dangerously-skip-permissions --model claude-opus-4-6 \
  "Прочитай HOW_TO_AGENTS.md и создай AGENTS.md, подходящий для этого проекта (на русском)"

# Рекомендуется: --dangerously-skip-permissions для непрерывной автономной настройки
# Рекомендуется: --model claude-opus-4-6 (или новее) для лучших результатов

# 2. Начните работу с созданными агентами
./ai-agency.sh

# 3. Начиная со следующих сессий — мгновенный старт работы!
```

> Этот документ предназначен **для чтения и понимания людьми**.
> Он объясняет, зачем существует инструкция для AI (HOW_TO_AGENTS.md), по какому принципу она работает
> и какую роль играет в вашем рабочем процессе разработки.

## Зачем нужна эта система?

### Проблема: AI каждый раз теряет память

```
 Сессия 1                   Сессия 2                   Сессия 3
┌──────────┐             ┌──────────┐             ┌──────────┐
│ AI читает │             │ AI снова  │             │ Опять с   │
│ весь код  │  Конец      │ читает    │  Конец      │ начала    │
│ (30 мин)  │ сессии     │ весь код  │ сессии     │ читает всё│
│ Начало    │ ──────→    │ (30 мин)  │ ──────→    │ (30 мин)  │
│ работы    │ Память     │ Начало    │ Память     │ Начало    │
│           │ стёрта!    │ работы    │ стёрта!    │ работы    │
└──────────┘             └──────────┘             └──────────┘
```

AI-агент забывает всё после завершения сессии.
Каждый раз он тратит время на изучение структуры проекта, анализ API и понимание соглашений.

### Решение: заранее создать AI «мозг»

```
 Начало сессии
┌──────────────────────────────────────────────────┐
│                                                  │
│  Чтение AGENTS.md (автоматически)                │
│       │                                          │
│       ▼                                          │
│  "Я — эксперт по бэкенду doppel-api"            │
│  "Соглашения: Conventional Commits, TypeScript strict" │
│  "Запрещено: изменять другие сервисы, хардкодить секреты" │
│       │                                          │
│       ▼                                          │
│  Загрузка файлов .ai-agents/context/ (5 сек)            │
│  "20 API, 15 сущностей, 8 событий — изучено"    │
│       │                                          │
│       ▼                                          │
│  Мгновенный старт работы!                        │
│                                                  │
└──────────────────────────────────────────────────┘
```

## Ключевой принцип: трёхуровневая структура

```
                    Ваш проект
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼

     ┌──────────┐  ┌──────────┐  ┌──────────┐
     │ AGENTS.md│  │.ai-agents/context│  │.ai-agents/skills│
     │          │  │          │  │          │
     │ Личность │  │ Знания   │  │ Действия │
     │ "Кто     │  │ "API     │  │ "При     │
     │  я?"     │  │  этого   │  │  разра-  │
     │          │  │  сервиса │  │  ботке   │
     │ + правила│  │  такой"  │  │  делай   │
     │ + права  │  │          │  │  так"    │
     │ + пути   │  │ + домен  │  │          │
     │          │  │ + модели │  │ + деплой │
     └──────────┘  └──────────┘  │ + ревью  │
                                 └──────────┘
      Точка входа   Хранилище     Стандарты
                    памяти        рабочих процессов
```

### 1. AGENTS.md — «Кто я?»

Файл **идентичности агента**, размещаемый в каждой директории.

```
проект/
├── AGENTS.md                  ← PM: лидер, координирующий всё
├── apps/
│   └── doppel-api/
│       └── AGENTS.md          ← Эксперт по API: отвечает только за этот сервис
├── infra/
│   ├── AGENTS.md              ← SRE: управление всей инфраструктурой
│   └── monitoring/
│       └── AGENTS.md          ← Эксперт по мониторингу
└── configs/
    └── AGENTS.md              ← Менеджер конфигураций
```

Это как **организационная структура команды**:
- PM видит общую картину и распределяет задачи
- Каждый участник глубоко разбирается только в своей области
- Задачи другой команды не выполняет, а передаёт запрос

### 2. `.ai-agents/context/` — «Что я знаю?»

Папка с **заранее систематизированными ключевыми знаниями**, чтобы AI не читал код каждый раз.

```
.ai-agents/context/
├── domain-overview.md     ← "Этот сервис отвечает за управление заказами..."
├── data-model.md          ← "Есть сущности Order, Payment, Delivery..."
├── api-spec.json          ← "POST /orders, GET /orders/{id}, ..."
└── event-spec.json        ← "Публикует событие order-created..."
```

**Аналогия:** как документы для онбординга нового сотрудника.
Один раз описать «чем занимается наша команда, какая структура БД, какие есть API» —
и не нужно объяснять каждый раз заново.

### 3. `.ai-agents/skills/` — «Как я работаю?»

**Руководства по рабочим процессам**, стандартизирующие повторяющиеся задачи.

```
.ai-agents/skills/
├── develop/SKILL.md       ← "Разработка фичи: анализ → проектирование → реализация → тесты → PR"
├── deploy/SKILL.md        ← "Деплой: тег → заявка → проверка"
└── review/SKILL.md        ← "Ревью: чек-лист по безопасности, производительности, тестам"
```

**Аналогия:** рабочие инструкции команды.
«При создании PR проверь этот чек-лист» — и AI тоже следует им.

## Как управлять глобальными правилами?

Используется **паттерн наследования**. Пишете в одном месте — автоматически применяется ниже по иерархии.

```
Корневой AGENTS.md ──────────────────────────────────────────
│ Global Conventions:
│  - Коммиты: Conventional Commits (feat:, fix:, chore:)
│  - PR: шаблон обязателен, минимум 1 ревьюер
│  - Ветки: feature/{ticket}-{desc}
│  - Код: TypeScript strict, single quotes
│
│     Авто-наследование              Авто-наследование
│     ┌──────────────────┐       ┌──────────────────┐
│     ▼                  │       ▼                  │
│  apps/api/AGENTS.md    │    infra/AGENTS.md       │
│  (только доп. правила)  │    (только доп. правила)  │
│  "Этот сервис           │    "При изменении         │
│   использует Python"    │     Helm values —         │
│     (вместо TypeScript) │     Ask First"           │
└─────────────────────────┴──────────────────────────
```

**Преимущества:**
- Хотите изменить правила коммитов? → Правите только в корне
- Добавляете новый сервис? → Глобальные правила применяются автоматически
- Нужно иначе для конкретного сервиса? → Переопределяете в его AGENTS.md

## Что писать, а что — нет?

Согласно исследованию ETH Zurich (2026), если документировать то, что AI может вывести самостоятельно,
**процент успеха снижается, а затраты возрастают на 20%**.

```
              Что писать                         Что НЕ писать
     ┌─────────────────────────┐     ┌─────────────────────────┐
     │                         │     │                         │
     │  "Коммиты в формате     │     │  "Исходники в папке src/"│
     │   feat:"                │     │  AI увидит через ls      │
     │  AI не может вывести    │     │                         │
     │                         │     │                         │
     │  "Прямой push в main    │     │  "React основан на       │
     │   запрещён"             │     │   компонентах"           │
     │  Правило команды,       │     │  Есть в документации     │
     │  в коде его нет         │     │                         │
     │                         │     │                         │
     │  "Перед деплоем нужно   │     │  "В этом файле 100 строк"│
     │   одобрение QA-команды" │     │  AI может прочитать сам  │
     │  Процесс, вывести       │     │                         │
     │  невозможно             │     │                         │
     └─────────────────────────┘     └─────────────────────────┘
          Писать в AGENTS.md               Не писать!
```

**Однако есть исключение:** «То, что AI может вывести, но каждый раз это слишком дорого»

```
  Пример: полный список API (нужно прочитать 20 файлов)
  Пример: схема связей моделей данных (разбросана по 10 файлам)
  Пример: граф вызовов между сервисами (нужно проверить и код, и инфраструктуру)

  → Такое заранее систематизируем в .ai-agents/context/!
  → В AGENTS.md указываем только путь: "смотри здесь"
```

## Скрипт запуска сессий

Когда все агенты настроены, можно выбрать нужного и сразу начать сессию.

```bash
$ ./ai-agency.sh

=== AI Agent Sessions ===
Project: /Users/nhn/src/doppel-deploy
Found: 8 agent(s)

  1) [PM] doppel-deploy
     Path: ./AGENTS.md
     PM-агент этого проекта. Понимает общую структуру и делегирует задачи.

  2) doppel-api
     Path: apps/dev/doppel-api/AGENTS.md
     Эксперт по управлению K8s-манифестами сервиса doppel-api.

  3) monitoring
     Path: infra/monitoring/AGENTS.md
     SRE-эксперт по стеку мониторинга Prometheus + Grafana.

  ...

Select agent (number): 2

=== AI Tool ===
  1) claude
  2) codex
  3) print

Select tool: 1

→ Сессия claude запущена в директории doppel-api
→ Агент автоматически загрузил AGENTS.md и .ai-agents/context/
→ Можно сразу работать!
```

**Параллельный запуск (tmux):**

```bash
$ ./ai-agency.sh --multi

Select agents: 1,2,3   # PM + API + мониторинг одновременно

→ Открыто 3 tmux-сессии
→ В каждом окне независимо работает свой агент
→ Переключение между окнами: Ctrl+B N
```

│       │                                                          │

## Список артефактов

Файлы, созданные этой системой, и их назначение:

| Файл | Для кого | Назначение |
|---|---|---|
| `HOW_TO_AGENTS.md` | AI | Мета-инструкция, которую агент читает и выполняет |
| `HOW_TO_AGENTS_PLAN.md` | Люди/AI | План проектирования (обоснование структуры) |
| `README.md` | Люди | Этот документ. Руководство для понимания людьми |
| `ai-agency.sh` | Люди | Лаунчер: выбор агента → запуск AI-сессии |
| `AGENTS.md` (в каждой директории) | AI | Идентичность агента + правила для директории |
| `.ai-agents/context/*.md/json` | AI | Заранее систематизированные доменные знания |
| `.ai-agents/skills/*/SKILL.md` | AI | Стандартизированные рабочие процессы |
| `.ai-agents/roles/*.md` | AI/Люди | Стратегии загрузки контекста по ролям |

## Ключевая аналогия

```
              Традиционная команда          Команда AI-агентов
              ─────────────────────        ─────────────────────
 Лидер        PM (человек)                 Корневой AGENTS.md (PM-агент)
 Участники    N разработчиков              AGENTS.md в каждой директории
 Онбординг    Confluence/Notion            .ai-agents/context/
 Инструкции   Вики команды                 .ai-agents/skills/
 Роли         Должности/R&R               .ai-agents/roles/
 Правила      Соглашения команды           Global Conventions (наследование)
 Приход       Приход в офис                Старт сессии → загрузка AGENTS.md
 Уход         Уход (память сохраняется)    Конец сессии (память стирается!)
 Следующий    Память есть                  Загрузка .ai-agents/context/ (восстановление памяти)
 день
```

**Ключевое отличие:** человек сохраняет память после ухода с работы, а AI забывает всё каждый раз.
Именно для этого существует `.ai-agents/context/` — он выполняет роль **долговременной памяти** AI.
